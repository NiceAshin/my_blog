# Linux 常用网络排查命令与网络状态诊断

在分布式系统和云原生时代，网络故障排查（如接口超时、TCP 连接积压、DNS 解析异常等）是后端开发与运维人员最为高频的技术场景。

Linux 系统内置了丰富的网络调试工具。熟练掌握 `ping`、`netstat`、`ss`、`curl`、`telnet` 等命令，能够帮助我们快速定位网络边界、诊断端口连通性并洞察复杂的 TCP 连接状态。

---

## 1. 核心网络命令字典

在进入故障排查之前，我们先来理清这五个命令的核心定位：

| 命令 | 核心作用定位 | 核心诊断维度 |
| :--- | :--- | :--- |
| **`ping`** | 检测主机双向连通性 | 基于 ICMP 协议，确认两端网络是否在物理上畅通 |
| **`telnet`** | 测试指定端口连通性 | 基于 TCP 协议，快速探测对端业务端口是否已开启监听 |
| **`curl`** | 发起 HTTP/HTTPS 请求 | 模拟应用层客户端，诊断 API 状态码及返回体 |
| **`netstat`** | 统计与显示网络状态 | 传统的网络连接审计工具，支持进程 PID 关联 |
| **`ss`** | 快速转储套接字状态 | 新一代高性能网络连接查看工具（比 netstat 快上百倍） |

---

## 2. 真实场景实战演练

### 场景 1：测试对端端口是否开放 (telnet vs curl)
当我们调用第三方接口出现 `Connection refused` 报错时，需要确认是本端网络阻断，还是对端服务未启动。

#### 示例 A：使用 `telnet` 探测端口
```bash
telnet 192.168.1.100 8080
```
- **输出分析**：
  - **成功连通**：终端屏幕变黑并显示 `Connected to 192.168.1.100`，说明 TCP 三次握手成功，对端端口已启动且防火墙未拦截。
  - **失败报错**：显示 `Connection timed out`，代表网络不通（可能防火墙拦截）；显示 `Connection refused`，代表网络通畅但对端目标端口并没有运行任何监听服务。

#### 示例 B：使用 `curl` 探测并提取 HTTP 状态码
有时我们只需要快速确认应用层接口状态：
```bash
# -I 代表仅获取 HTTP 响应头；-s 代表静默模式不显示进度条
curl -Is http://192.168.1.100:8080/health
```
- **输出示例**：
  ```text
  HTTP/1.1 200 OK
  Content-Type: application/json;charset=UTF-8
  Date: Wed, 10 Jun 2026 02:37:28 GMT
  ```

### 场景 2：查看服务器 TCP 连接积压 (netstat vs ss)
当我们的 Spring Boot 或 Go 服务负载过高时，需要检查底层的网络套接字状况。

#### 示例 A：使用 `ss` 快速获取套接字摘要
当系统连接数高达数万时，`netstat` 会通过读取 `/proc/net/` 导致严重卡顿。而 `ss` 绕过内核直接读取，速度极快：
```bash
# -s 打印当前套接字状态的简明统计汇总
ss -s
```
- **输出示例**：
  ```text
  Total: 1254
  TCP:   320 (estab 145, closed 12, orphaned 0, timewait 45)
  Transport Total     IP      IPv6
  RAW	      0        0       0
  UDP	      15       10      5
  TCP	      332      300     32
  ```

#### 示例 B：定位特定端口上正在监听的进程 (netstat -tulnp)
我们需要确认到底是什么进程在霸占系统的 `8080` 端口：
```bash
# -t (TCP), -u (UDP), -l (Listening), -n (Numeric), -p (Show PID/Program)
netstat -tulnp | grep :8080
```
- **输出示例**：
  ```text
  tcp6       0      0 :::8080                 :::*                    LISTEN      31205/java
  ```
  - **分析**：这说明是一个 PID 为 `31205` 的 `java` 进程在监听 IPv6 端口 8080。

### 场景 3：诊断网络双向丢包与时延 (ping)
`ping` 基于 ICMP 报文检测网络可达性。

#### 命令规范：
```bash
# -c 5 表示仅发送 5 次报文；-i 0.2 表示每隔 0.2 秒发送一次
ping -c 5 -i 0.2 8.8.8.8
```
- **输出示例**：
  ```text
  5 packets transmitted, 5 received, 0% packet loss, time 1002ms
  rtt min/avg/max/mdev = 12.115/15.240/18.450/2.105 ms
  ```
  - **关键指标分析**：
    - `0% packet loss`：丢包率为 0，证明链路网络非常健康。
    - `avg 15.240 ms`：双向往返平均时延为 15.24 毫秒。如果该值飙升至数百毫秒，代表路由节点拥堵。

## 3. 网络故障诊断标准三步法

当在生产环境中遭遇“接口无法连通”时，推荐按照以下漏斗模型逐步排查：

1. **第一步（网络层连通）**：`ping <对端IP>`。确认两台物理服务器之间路由正常。
2. **第二步（传输层连通）**：`telnet <对端IP> <目标端口>`。确认对端对应的安全组/防火墙已对您放行，且服务正在监听。
3. **第三步（应用层连通）**：`curl -v <对端API网址>`。通过输出的详细 HTTP 报文，观察握手、重定向与返回的状态码。
