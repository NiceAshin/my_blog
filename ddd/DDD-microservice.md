# DDD 指导微服务拆分

微服务拆分是架构设计中最关键的决策之一。拆分过粗，单体变"分布式单体"；拆分过细，运维成本飙升。DDD 的限界上下文为微服务拆分提供了天然的理论指导。

---

## 1. 微服务拆分的困境

### 1.1 常见错误

```
┌─────────────────────────────────────────────────────────────┐
│                    错误的拆分方式                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ❌ 按技术层拆分                                            │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│   │ 前端服务  │  │ 后端服务  │  │ 数据服务  │                │
│   └──────────┘  └──────────┘  └──────────┘                 │
│   问题：业务逻辑分散，修改跨多个服务                          │
│                                                             │
│   ❌ 按实体拆分                                              │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│   │ 用户服务  │  │ 订单服务  │  │ 商品服务  │                │
│   └──────────┘  └──────────┘  └──────────┘                 │
│   问题：一个业务操作涉及多个服务，事务复杂                    │
│                                                             │
│   ❌ 拆分过细                                                │
│   ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐            │
│   │认证   ││授权   ││用户   ││配置   ││日志   │            │
│   └──────┘ └──────┘ └──────┘ └──────┘ └──────┘            │
│   问题：服务数量爆炸，运维噩梦                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 正确的拆分原则

```
┌─────────────────────────────────────────────────────────────┐
│                    正确的拆分原则                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ✅ 按业务能力拆分（Business Capability）                    │
│      一个服务对应一个业务能力                                │
│                                                             │
│   ✅ 高内聚、松耦合                                          │
│      业务逻辑集中，服务间依赖少                              │
│                                                             │
│   ✅ 单一职责                                                │
│      每个服务做好一件事                                      │
│                                                             │
│   ✅ 独立部署                                                │
│      服务可以独立开发、测试、部署                            │
│                                                             │
│   ✅ 团队对齐（康威定律）                                     │
│      服务边界与团队边界对齐                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 2. 限界上下文即服务边界

### 2.1 核心映射关系

```
┌─────────────────────────────────────────────────────────────┐
│               DDD 概念 → 微服务映射                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   限界上下文 (Bounded Context)  ──────▶  微服务              │
│                                                             │
│   子域 (Subdomain)              ──────▶  服务定位            │
│   - 核心域                      ──────▶  核心服务，重点投入   │
│   - 支撑域                      ──────▶  支撑服务，适度投入   │
│   - 通用域                      ──────▶  通用服务，可采购     │
│                                                             │
│   聚合 (Aggregate)              ──────▶  服务内部单元        │
│                                                             │
│   领域事件 (Domain Event)       ──────▶  服务间通信          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 拆分决策矩阵

| 子域类型 | 业务价值 | 变化频率 | 拆分策略 |
|----------|----------|----------|----------|
| 核心域 | 高 | 高 | 独立服务，持续迭代 |
| 支撑域 | 中 | 中 | 独立服务或合并 |
| 通用域 | 低 | 低 | 可采购/开源方案 |

## 3. 拆分方法论

### 3.1 事件风暴工作坊

```
┌─────────────────────────────────────────────────────────────┐
│                   事件风暴步骤                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 识别领域事件                                        │
│  ┌─────────────────────────────────────────────────────┐   │

│  Step 2: 识别命令和触发者                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ [用户] 下单 → [系统] 创建订单 → 订单已创建            │   │

│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  Step 4: 划分边界（关键步骤）                                │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│  │ 订单上下文 │  │ 支付上下文 │  │ 物流上下文 │                │
│  │          │  │          │  │          │                 │
│  │ 订单聚合  │  │ 支付聚合  │  │ 运单聚合  │                 │
│  └──────────┘  └──────────┘  └──────────┘                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 5. 服务间通信

### 5.1 通信模式选择

```
┌─────────────────────────────────────────────────────────────┐
│                   服务间通信模式                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   同步通信 (RPC/HTTP)                                        │
│   ┌─────────────────────────────────────────────────────┐   │
│   │ 适用场景：                                             │   │
│   │ - 需要实时响应                                        │   │
│   │ - 数据强一致性要求                                    │   │
│   │ - 查询类操作                                          │   │
│   │                                                       │   │
│   │ 示例：订单服务 → 商品服务（查询商品信息）              │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   异步通信 (消息队列)                                        │
│   ┌─────────────────────────────────────────────────────┐   │
│   │ 适用场景：                                             │   │
│   │ - 跨服务事务                                          │   │
│   │ - 解耦、削峰填谷                                      │   │
│   │ - 最终一致性可接受                                    │   │
│   │                                                       │   │
│   │ 示例：订单服务 → 物流服务（订单已支付事件）            │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 事件驱动架构

```java
// 订单服务发布事件
@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final EventPublisher eventPublisher;

    public void payOrder(OrderId orderId, PaymentId paymentId) {
        Order order = orderRepository.findById(orderId);
        order.pay(paymentId);
        orderRepository.save(order);

        // 发布事件
        eventPublisher.publish(new OrderPaidEvent(orderId, paymentId));
    }
}

// 物流服务订阅事件
@Component
public class ShippingEventHandler {

    private final ShippingService shippingService;

    @KafkaListener(topics = "order-paid")
    public void onOrderPaid(OrderPaidEvent event) {
        // 创建运单
        shippingService.createShipment(event.getOrderId());
    }
}
```

### 5.3 Saga 分布式事务

```
┌─────────────────────────────────────────────────────────────┐
│                    Saga 编排模式                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   订单创建流程 Saga:                                         │
│                                                             │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐│
│   │ 创建订单 │───▶│ 冻结库存 │───▶│ 创建支付 │───▶│ 确认订单 ││
│   └─────────┘    └─────────┘    └─────────┘    └─────────┘│
│        │              │              │                     │
│        │ 失败         │ 失败         │ 失败                │
│        ▼              ▼              ▼                     │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐               │
│   │ 取消订单 │◀───│ 释放库存 │◀───│ 取消支付 │               │
│   └─────────┘    └─────────┘    └─────────┘               │
│       补偿          补偿          补偿                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```java
// Saga 编排器
public class CreateOrderSaga {

    private final OrderService orderService;
    private final InventoryService inventoryService;
    private final PaymentService paymentService;

    public void execute(CreateOrderCommand command) {
        try {
            // Step 1: 创建订单
            Order order = orderService.create(command);

            // Step 2: 冻结库存
            inventoryService.freeze(order.getItems());

            // Step 3: 发起支付
            paymentService.createPayment(order);

            // Step 4: 确认订单
            orderService.confirm(order.getId());

        } catch (Exception e) {
            // 执行补偿
            compensate(command);
            throw e;
        }
    }

    private void compensate(CreateOrderCommand command) {
        // 释放库存
        inventoryService.release(command.getItems());
        // 取消订单
        orderService.cancel(command.getOrderId());
    }
}
```

## 6. 数据一致性策略

### 6.1 数据库分离原则

```
┌─────────────────────────────────────────────────────────────┐
│                   数据库分离原则                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ✅ 每个服务有独立的数据库                                   │
│   ✅ 服务间不共享数据库表                                     │
│   ✅ 通过 API 或事件共享数据                                  │
│   ✅ 各服务可以选择不同的存储技术                             │
│                                                             │
│   ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│   │  订单服务    │  │  商品服务    │  │  用户服务    │        │
│   │             │  │             │  │             │        │
│   │  ┌───────┐  │  │  ┌───────┐  │  │  ┌───────┐  │        │
│   │  │ MySQL │  │  │  │ MySQL │  │  │  │ MySQL │  │        │
│   │  └───────┘  │  │  └───────┘  │  │  └───────┘  │        │
│   └─────────────┘  └─────────────┘  └─────────────┘        │
│        订单库            商品库            用户库            │
│                                                             │
│   ❌ 错误：服务共享数据库                                     │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  订单服务 ─┐                                         │  │
│   │            ├──▶ 共享数据库（紧耦合，无法独立演进）     │  │
│   │  商品服务 ─┘                                         │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 一致性模式对比

| 模式 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| 2PC | 强一致 | 低 | 高 | 传统分布式事务 |
| TCC | 强一致 | 中 | 高 | 金融、交易 |
| Saga | 最终一致 | 高 | 中 | 长流程业务 |
| 本地消息表 | 最终一致 | 高 | 低 | 简单场景 |
| 事务消息 | 最终一致 | 高 | 中 | 异步解耦 |

### 7.2 拆分优先级

```
优先拆分顺序：

1. 通用域（低风险）
   └── 用户服务、通知服务
   └── 可以直接使用开源方案

2. 边界清晰的支撑域（中风险）
   └── 支付服务、物流服务
   └── 业务相对独立

3. 核心域（高风险，最后拆分）
   └── 订单服务、商品服务
   └── 需要充分理解业务后再拆
```

## 8. 常见问题与解决方案

### 8.1 跨服务查询

**问题**：查询需要聚合多个服务的数据

**解决方案**：CQRS + 读模型投影

```java
// 为查询创建专用读模型
@Entity
@Table(name = "order_query_view")
public class OrderQueryView {

    @Id
    private String orderId;
    private String orderStatus;
    private String customerName;      // 冗余用户数据
    private String customerPhone;     // 冗余用户数据
    private List<OrderItemView> items;
    private BigDecimal totalAmount;
    private String shippingAddress;
    private Instant createdAt;

    // 针对查询优化，包含所有需要展示的字段
}

// 事件投影器保持读模型更新
@Component
public class OrderQueryProjector {

    @EventHandler
    public void on(OrderCreated event) {
        // 从用户服务获取用户信息
        UserInfo user = userServiceClient.getUser(event.getCustomerId());

        OrderQueryView view = new OrderQueryView();
        view.setOrderId(event.getOrderId());
        view.setCustomerName(user.getName());
        view.setCustomerPhone(user.getPhone());
        // ...

        repository.save(view);
    }
}
```

### 8.2 分布式事务

**问题**：跨服务操作需要保证一致性

**解决方案**：Saga + 补偿机制

```
┌─────────────────────────────────────────────────────────────┐
│                    Saga 示例                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   下单流程：                                                 │
│                                                             │
│   正向流程：                                                 │
│   订单服务 ──创建订单──▶ 商品服务 ──冻结库存──▶              │
│   支付服务 ──创建支付──▶ 订单服务 ──确认订单──▶ 完成         │
│                                                             │
│   补偿流程（任一步骤失败）：                                  │
│   订单服务 ◀──取消订单── 商品服务 ◀──释放库存──             │
│   支付服务 ◀──取消支付──                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 8.3 服务发现

**问题**：服务如何找到其他服务的地址

**解决方案**：服务注册中心

```yaml
# application.yml (Spring Cloud)
spring:
  application:
    name: order-service
  cloud:
    nacos:
      discovery:
        server-addr: nacos:8848

# 服务调用
@FeignClient(name = "product-service")
public interface ProductClient {
    @GetMapping("/products/{id}")
    ProductDTO getProduct(@PathVariable String id);
}
```

## 9. 小结

DDD 指导微服务拆分的核心要点：

| 步骤 | 内容 |
|------|------|
| 业务分析 | 识别核心域、支撑域、通用域 |
| 边界划分 | 事件风暴识别限界上下文 |
| 服务定义 | 限界上下文映射为微服务 |
| 通信设计 | 定义 API 和事件契约 |
| 数据分离 | 每个服务独立数据库 |
| 渐进演进 | 从模块化单体逐步拆分 |

**记住**：限界上下文是微服务拆分的天然边界，先充分理解业务，再动手拆分。
