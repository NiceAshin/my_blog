    # DDD 战略设计：限界上下文与子域

    战略设计侧重于宏观层面的系统拆分和组织架构，解决"系统怎么拆"的核心问题。通过合理的边界划分，可以有效控制系统复杂度，为微服务架构奠定基础。

    # 1. 子域 (Subdomain)

    子域是对业务问题的划分，从业务视角识别系统的不同部分。

    # 1.1 子域分类

    ┌─────────────────────────────────────────────────────────────┐
│                      业务系统                                │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────────┐   │
│  │                 核心域 (Core Domain)                  │   │
│  │     业务核心竞争力，必须自主开发，持续优化              │   │
│  │     例：电商的商品推荐算法、外卖的配送调度              │   │
│  └─────────────────────────────────────────────────────┘   │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────────────┐   ┌─────────────────────────────┐ │
│  │   支撑域 (Supporting)│   │   通用域 (Generic Domain)    │ │
│  │   业务必需但非核心   │   │   各行业通用功能             │ │
│  │   例：电商的营销    │   │   例：支付、权限、消息通知    │ │
│  │       外卖的骑手管理 │   │       可采购成熟方案         │ │
│  └─────────────────────┘   └─────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘

    # 1.2 子域识别方法

    事件风暴 (Event Storming) 是最常用的子域识别方法：

    ┌──────────────────────────────────────────────────────────────┐
│                      事件风暴工作坊                            │
├──────────────────────────────────────────────────────────────┤
│  1. 遍历领域事件（橙色便利贴）                                  │
│     订单已创建 → 订单已支付 → 商品已发货 → 订单已完成            │
│                                                              │
│  2. 识别命令（蓝色便利贴）                                      │
│     创建订单    确认支付    发货        确认收货                │
│                                                              │
│  3. 找出聚合（黄色便利贴）                                      │
│     Order       Payment    Shipment                          │
│                                                              │
│  4. 划分边界                                                  │
│     ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│     │ 订单子域  │  │ 支付子域  │  │ 物流子域  │                │
│     └──────────┘  └──────────┘  └──────────┘                 │
└──────────────────────────────────────────────────────────────┘

    # 1.3 便利贴颜色约定

    颜色 元素 说明
    🟠 橙色 领域事件 已发生的事实，用过去时态
    🔵 蓝色 命令 触发事件的动作
    🟡 黄色 聚合 处理命令的一组对象
    🟢 绿色 读模型 查询用的数据投影
    🟣 紫色 策略 事件触发的后续动作
    ⬜ 白色 外部系统 系统边界外的服务

    # 2. 限界上下文 (Bounded Context)

    限界上下文是模型的语义边界，在该边界内，每个术语都有明确的含义。

    # 2.1 为什么需要限界上下文？

    同一个词在不同上下文中有不同含义：

    ┌─────────────────────────────────────────────────────────────┐
│                     "用户" 的不同含义                         │
├─────────────────────┬───────────────────────────────────────┤
│     上下文          │              含义                      │
├─────────────────────┼───────────────────────────────────────┤
│   用户上下文         │  注册信息：用户名、密码、邮箱            │
├─────────────────────┼───────────────────────────────────────┤
│   订单上下文         │  购买者：收货地址、联系方式              │
├─────────────────────┼───────────────────────────────────────┤
│   商品上下文         │  浏览者：浏览历史、偏好标签              │
├─────────────────────┼───────────────────────────────────────┤
│   支付上下文         │  付款人：支付方式、银行卡信息            │
└─────────────────────┴───────────────────────────────────────┘

    # 2.2 限界上下文划分原则

    ┌─────────────────────────────────────────────────────────────┐
│                     划分原则                                  │
├─────────────────────────────────────────────────────────────┤
│  ✅ 业务语义一致                                              │
│     同一上下文内的术语含义统一                                 │
│                                                              │
│  ✅ 功能内聚                                                  │
│     相关功能放在一起，减少跨上下文调用                         │
│                                                              │
│  ✅ 数据独立                                                  │
│     每个上下文管理自己的数据存储                               │
│                                                              │
│  ✅ 团队对齐                                                  │
│     一个上下文对应一个开发团队（康威定律）                      │
│                                                              │
│  ⚠️ 边界不要过细                                              │
│     避免过度拆分导致分布式事务问题                             │
└─────────────────────────────────────────────────────────────┘

    # 2.3 子域与限界上下文的关系

    ┌──────────────────────────────────────────────────────────────┐
│                                                              │
│    子域（问题空间）          限界上下文（解空间）              │
│                                                              │
│   ┌─────────────┐           ┌─────────────┐                 │

│   实际情况：可能多个子域合并到一个上下文                       │
│                                                              │
└──────────────────────────────────────────────────────────────┘

    # 3. 上下文映射 (Context Mapping)

    上下文映射描述不同限界上下文之间的协作关系。

    # 3.1 映射模式

    ┌──────────────────────────────────────────────────────────────┐
│                    上下文映射关系                             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────┐                      ┌─────────────┐        │
│  │   上游      │    共享内核           │   下游      │        │
│  │   (U)      │◄──────────────────────│   (D)      │        │
│  └─────────────┘                      └─────────────┘        │
│       Shared Kernel                                         │
│       共享一部分模型代码                                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────┐                      ┌─────────────┐        │
│  │   上游      │    开放主机服务        │   下游      │        │
│  │   (U)      │──────────────────────▶│   (D)      │        │
│  └─────────────┘                      └─────────────┘        │
│       Open Host Service (OHS)                               │
│       提供标准 API 接口                                       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────┐                      ┌─────────────┐        │
│  │   上游      │    遵奉者             │   下游      │        │
│  │   (U)      │──────────────────────▶│   (D)      │        │
│  └─────────────┘                      └─────────────┘        │
│       Conformist                                            │
│       下游完全遵循上游模型                                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────┐                      ┌─────────────┐        │
│  │   上游      │    防腐层             │   下游      │        │
│  │   (U)      │──────────────────────▶│   (D)      │        │
│  └─────────────┘       [ACL]          └─────────────┘        │
│       Anti-Corruption Layer                                │
│       隔离上游模型，转换为本地模型                             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────┐                      ┌─────────────┐        │
│  │   上游      │    发布语言           │   下游      │        │
│  │   (U)      │──────────────────────▶│   (D)      │        │
│  └─────────────┘       [PL]           └─────────────┘        │
│       Published Language                                    │
│       使用标准的交换格式（如 JSON Schema）                     │
└──────────────────────────────────────────────────────────────┘

                        ┌─────────────────┐
                    │    用户上下文    │
                    │   User Context  │
                    └────────┬────────┘

┌─────────────┐        ┌─────────────┐        ┌─────────────┐

    # 3.3 防腐层实现示例

    // 订单上下文中的防腐层
@Component
public class ProductAdapter {

    private final ProductClient productClient;

    // 外部商品 DTO
    public ProductAdapter(ProductClient productClient) {
        this.productClient = productClient;
    }

    // 转换为订单上下文的本地模型
    public ProductInfo getProductInfo(ProductId productId) {
        // 调用外部服务
        ProductDTO external = productClient.getById(productId.getValue());

        // 防腐层：转换为本地模型
        return ProductInfo.builder()
            .productId(new ProductId(external.getId()))
            .productName(external.getTitle())  // 字段名映射
            .unitPrice(Money.of(external.getPrice(), "CNY"))
            .stock(external.getInventory())    // 语义转换
            .build();
    }
}

    # 4. 实践：电商系统战略设计

    # 4.1 识别子域

    子域 类型 核心能力
    商品子域 支撑域 商品管理、库存管理、价格策略
    订单子域 核心域 下单、改单、取消、退款
    支付子域 通用域 支付渠道、支付流水、对账
    用户子域 通用域 注册、登录、权限
    营销子域 支撑域 优惠券、活动、促销
    物流子域 支撑域 发货、配送、签收

    # 4.2 划分限界上下文

│  商品上下文  │  订单上下文  │  支付上下文  │    用户上下文    │
│   Product   │    Order    │   Payment   │      User       │
│   Context   │   Context   │   Context   │     Context     │
├─────────────┼─────────────┼─────────────┼─────────────────┤
│  营销上下文  │  物流上下文  │                             │
│ Marketing  │  Shipping   │                             │

    private String userId;
    private List<OrderItemDTO> items;

    # 5. 常见问题

    # 5.1 上下文边界如何确定？

    方法 适用场景
    事件风暴 复杂业务、团队协作探索
    用例分析 相对简单的业务系统
    领域专家访谈 业务专家参与度高的项目
    现有系统分析 遗留系统重构

    # 5.2 上下文之间如何通信？

    通信方式 适用场景 特点
    同步 RPC 需要实时响应 强耦合、低延迟
    异步消息 最终一致性可接受 解耦、高可用
    事件驱动 业务解耦要求高 最终一致、可扩展

    # 5.3 数据一致性如何保证？

    ┌─────────────────────────────────────────────────────────────┐
│              分布式事务解决方案                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. Saga 模式（推荐）                                        │
│     每个上下文本地事务 + 补偿机制                             │
│                                                             │
│  2. 事件驱动 + 最终一致性                                     │
│     发布领域事件，下游订阅处理                                │
│                                                             │
│  3. TCC（Try-Confirm-Cancel）                               │
│     强一致性要求高的场景                                      │
│                                                             │
│  4. 本地消息表                                               │
│     简单可靠的最终一致性方案                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘

    # 6. 小结

    产出物 说明
