# 领域驱动设计 (DDD) 核心思想

领域驱动设计（Domain-Driven Design，简称 DDD）是一种通过将实现连接到持续进化的模型来满足复杂需求的软件开发方法。由 Eric Evans 在 2003 年的著作《Domain-Driven Design: Tackling Complexity in the Heart of Software》中首次系统阐述。

---

## 1. 为什么需要 DDD？

### 1.1 传统开发模式的困境

在传统的三层架构中，业务逻辑往往散落在各处：

```
┌─────────────────────────────────────────────────┐
│                   表现层                          │
│              (Controller/API)                    │
├─────────────────────────────────────────────────┤
│                   业务层                          │
│     (Service - 业务逻辑混杂着事务、权限等)          │
├─────────────────────────────────────────────────┤
│                   数据层                          │
│         (Repository/DAO - 直接操作表结构)          │
└─────────────────────────────────────────────────┘
```

**典型问题**：
- **贫血模型**：对象只有 getter/setter，没有行为
- **业务散落**：规则分散在 Service、Controller、甚至 SQL 中
- **难以维护**：修改一个业务规则需要改动多处
- **沟通成本高**：技术人员和业务人员使用不同的语言

### 1.2 DDD 的解决思路

DDD 通过以下方式解决上述问题：

| 问题 | DDD 方案 |
|------|----------|
| 贫血模型 | 充血模型：实体包含行为 |
| 业务散落 | 聚合边界：一致性边界内封装规则 |
| 难以维护 | 领域模型：统一修改入口 |
| 沟通成本 | 统一语言：技术与业务使用相同术语 |

## 2. DDD 核心价值

### 2.1 统一语言 (Ubiquitous Language)

**统一语言是 DDD 最重要的实践**，它要求开发团队和业务专家使用相同的术语描述领域概念。

**示例**：电商订单场景

```
❌ 技术语言：
"把 order_status 字段从 0 改成 1，然后插入一条 order_log 记录"

✅ 统一语言：
"确认订单，状态变为已支付，记录支付事件"
```

**实践建议**：
- 代码中的类名、方法名直接使用业务术语
- 文档、注释与代码保持一致
- 在代码评审中检查术语使用

### 2.2 战略设计

从宏观层面划分系统边界，解决"系统怎么拆"的问题：

```
┌──────────────────────────────────────────────────────────────┐
│                        电商系统                               │
├────────────────┬────────────────┬───────────────────────────┤
│   订单上下文    │   商品上下文    │       支付上下文           │
│ Order Context  │ Product Context│     Payment Context       │
├────────────────┼────────────────┼───────────────────────────┤
│   库存上下文    │   用户上下文    │       物流上下文           │
│Inventory Ctx   │  User Context  │    Shipping Context       │
└────────────────┴────────────────┴───────────────────────────┘
```

### 2.3 战术设计

从微观层面组织代码结构，解决"代码怎么写"的问题：

```
┌─────────────────────────────────────────────────────────────┐
│                      聚合 (Aggregate)                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              聚合根 (Aggregate Root)                  │   │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────────────────┐  │   │
│  │  │  实体   │  │ 值对象  │  │     领域服务        │  │   │
│  │  │ Entity  │  │Value Obj│  │  Domain Service    │  │   │
│  │  └─────────┘  └─────────┘  └─────────────────────┘  │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

## 3. DDD 分层架构

### 3.1 四层架构

```
┌──────────────────────────────────────────────────────────────┐
│                    用户界面层 (UI Layer)                      │
│         Controller / Facade / DTO                            │
├──────────────────────────────────────────────────────────────┤
│                    应用层 (Application Layer)                 │
│         Application Service / Command / Query                │
├──────────────────────────────────────────────────────────────┤
│                    领域层 (Domain Layer)                      │
│         Aggregate / Entity / Value Object / Domain Service   │
├──────────────────────────────────────────────────────────────┤
│                    基础设施层 (Infrastructure Layer)           │
│         Repository Implementation / MQ / External Service    │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 各层职责

| 层次 | 职责 | 典型组件 |
|------|------|----------|
| 用户界面层 | 接收请求、参数校验、响应格式化 | Controller, DTO,Assembler |
| 应用层 | 编排业务流程、事务管理 | Application Service, Command Handler |
| 领域层 | 核心业务逻辑、领域规则 | Aggregate, Entity, Value Object, Domain Service |
| 基础设施层 | 技术实现、外部资源访问 | RepositoryImpl, MessageProducer, ExternalClient |

### 3.3 依赖方向

**核心原则**：依赖指向领域层，基础设施层依赖领域层接口。

```java
// 领域层定义接口
public interface OrderRepository {
    Order findById(OrderId id);
    void save(Order order);
}

// 基础设施层实现接口
@Repository
public class JpaOrderRepository implements OrderRepository {
    // 具体的 JPA 实现
}
```

## 4. DDD 与微服务

### 4.1 限界上下文与微服务

DDD 的限界上下文天然适合作为微服务的拆分边界：

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   订单服务       │     │   商品服务       │     │   支付服务       │
│ Order Service   │────▶│ Product Service │────▶│ Payment Service │
│                 │     │                 │     │                 │
│ 订单上下文       │     │ 商品上下文       │     │ 支付上下文       │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

### 4.2 微服务拆分原则

| 原则 | 说明 |
|------|------|
| 单一职责 | 每个服务对应一个限界上下文 |
| 高内聚 | 聚合内的对象应该在一起 |
| 松耦合 | 上下文之间通过接口或事件通信 |
| 独立部署 | 每个服务可以独立开发、测试、部署 |

## 5. 快速上手清单

### 5.1 什么时候用 DDD？

| 场景 | 是否推荐 DDD |
|------|-------------|
| 业务逻辑复杂的系统 | ✅ 强烈推荐 |
| 需要长期演进的产品 | ✅ 推荐 |
| 简单 CRUD 应用 | ❌ 过度设计 |
| 原型验证阶段 | ❌ 先快速迭代 |
| 团队不熟悉 DDD | ⚠️ 先小范围试点 |

### 5.2 实施路线图

```
第1阶段：建立统一语言
    ├── 梳理业务术语表
    ├── 建立领域词典
    └── 代码命名规范

第2阶段：战略设计
    ├── 识别子域
    ├── 划分限界上下文
    └── 建立上下文映射

第3阶段：战术设计
    ├── 识别聚合
    ├── 设计实体和值对象
    └── 定义领域服务

第4阶段：持续演进
    ├── 重构优化
    ├── 模型迭代
    └── 知识沉淀
```
