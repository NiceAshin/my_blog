    # 从提示词工程到上下文工程：让大模型稳定、可控、可复用

    预计阅读：17 分钟
    适合读者：正在做 AI 应用、知识库、智能助手、代码助手，希望解决“同一个模型一会儿聪明一会儿失常”的团队。

    过去两年，很多团队把大模型应用优化，简化成“改 Prompt”。模型答错了，就补一句要求；输出不稳了，就加一条格式限制；效果漂移了，就再堆几条例子。短期看，这种方法确实有效，但一旦系统复杂起来，单纯的 Prompt 调整会越来越像“修补一堵随时漏风的墙”。

    原因在于：影响模型效果的，远不止 Prompt 文本本身。模型最终表现，受到系统角色、任务目标、外部知识、历史状态、工具结果、输出协议、评测反馈等多种上下文共同影响。也正因为如此，越来越多团队开始从“提示词工程”走向“上下文工程（Context Engineering）”。

    本文想回答的问题是：为什么只改 Prompt 已经不够？什么是上下文工程？怎样把上下文设计成一种可维护、可组合、可评测的工程能力？

    # 1. 为什么“只会写 Prompt”已经不够了

    在简单场景里，一个好 Prompt 当然重要。但当系统进入真实业务后，模型的输入不再只是一个提示词，而是一个复杂的上下文包。这个上下文包可能包括：

    • 系统角色与安全规则；
    • 当前任务目标与业务约束；
    • 用户历史对话；
    • 检索得到的文档片段；
    • 工具调用结果；
    • 输出格式定义；
    • 示例样本；
    • 组织术语与风格偏好。

    如果这些信息组织混乱，再漂亮的 Prompt 也救不了系统。换句话说，Prompt 是上下文的一部分，但不是全部。

    # 2. 什么是上下文工程

    上下文工程可以理解为：围绕模型当前任务，把最相关、最可信、最可执行的信息，以结构化方式送入模型，并对其生命周期进行治理。

    这里有三个关键词：

    • 相关：不是越多越好，而是越贴近当前任务越好；
    • 结构化：不是把信息一股脑堆进去，而是分层组织；
    • 治理：不是一次性拼接，而是可更新、可观测、可评估。

    因此，上下文工程既包括 Prompt 模板设计，也包括知识检索、消息裁剪、记忆管理、工具输出压缩、格式约束和评测回路。

    # 3. 一个实用的上下文分层模型

    在工程实践中，我更推荐把上下文拆成五层：

    # 3.1 规则层

    定义模型身份、边界、禁止事项、输出协议。比如：不得编造来源、不得执行未授权动作、必须输出 JSON、遇到高风险问题必须建议人工确认。

    # 3.2 任务层

    定义这一次任务要做什么、目标用户是谁、成功标准是什么。任务层回答的是“这次到底要完成什么”。

    # 3.3 证据层

    提供事实依据，例如检索到的文档、数据库查询结果、工具响应。证据层回答的是“你凭什么这么说”。

    # 3.4 过程层

    保存中间状态，例如已完成步骤、已尝试方案、尚未解决的问题。过程层回答的是“已经做到哪一步了”。

    # 3.5 表达层

    定义输出风格、语言、长度、格式和模板。表达层回答的是“结果应该怎样呈现给人看”。

    把这五层分开后，你会明显感觉系统更好维护，因为每一层都可以独立优化。

    # 4. 好 Prompt 的本质：降低歧义，不是堆更多字

    很多 Prompt 写得很长，但并不有效。问题不在于长短，而在于是否减少了歧义。一个高质量 Prompt，通常会明确以下内容：

    • 你是谁；
    • 你要完成什么任务；
    • 你可以使用哪些信息与工具；
    • 你不能做什么；
    • 你应该按什么格式输出；
    • 信息不足时要怎么办。

    例如，不要只写“请总结这份文档”，而是写成：

    你是一名面向技术经理的文档分析助手。
请阅读提供的材料，输出：
1. 三条核心结论；
2. 两个风险点；
3. 需要继续确认的问题。
如果材料不足，请明确指出缺失信息。
输出格式使用 Markdown 列表。
    1
    2
    3
    4
    5
    6
    7

    这段 Prompt 并不复杂，但它显著减少了任务目标和输出格式的不确定性。

    # 5. 上下文预算：最重要的信息，必须占据注意力中心

    上下文窗口变大，并不意味着你可以无脑塞满。模型的注意力是稀缺资源，信息越多，噪声越容易稀释重点。

    一个常见错误是：

    • 前面放很长的系统规则；
    • 中间塞进全部聊天记录；
    • 后面附上十几段检索材料；
    • 最后再放真正要完成的任务。

    结果是模型看起来拥有很多信息，实际上却抓不住重点。更好的方法是：

    • 让任务目标靠前且明确；
    • 只保留与当前问题强相关的历史；
    • 对工具输出先做压缩摘要；
    • 对检索材料先重排再注入；
    • 把长期规则抽成稳定模板，不要每次重写。

    # 6. Few-shot 示例：不是越多越好，而是越贴近失败点越好

    示例很有用，但它的真正作用不是“告诉模型更多背景”，而是校准模型对任务模式的理解。因此，Few-shot 的价值来自代表性，而不是数量。

    一个实用策略是围绕“最容易失败的点”准备示例，例如：

    • 输出格式老是不稳，就提供结构化输入输出对；
    • 模型总爱过度发挥，就提供“信息不足时拒答”的例子；
    • 模型常误解术语，就提供业务词汇的正反例。

    示例应该像测试用例，而不是像宣传文案。它的目标是校准行为，而不是展示文笔。

    # 7. 检索内容如何进入上下文

    在 RAG 或 Agent 系统里，真正让效果拉开差距的，常常不是有没有检索，而是检索结果以什么方式进入上下文。

    推荐遵循三个原则：

    # 7.1 先筛选，再注入

    不要把 top-k 结果原封不动塞给模型。先做重排、去重和摘要，把最有价值的信息留下。

    # 7.2 明确来源边界

    给每段证据附上来源、标题、时间、权限，有助于模型在回答时保持边界感，也利于最终输出引用。

    # 7.3 区分“事实材料”和“推理要求”

    上下文里最好显式区分“这是材料”“这是任务要求”“这是输出格式”。信息边界越清楚，模型越不容易混淆。

    # 8. 记忆管理：长期记忆不是聊天记录堆积

    不少团队把“记忆”理解成保存全部历史消息。但对模型来说，历史越多并不一定越有帮助。真正有用的长期记忆，应该是经整理后的稳定偏好与事实。

    比如：

    • 用户偏好中文回答；
    • 团队内部习惯把“发布回滚”叫作“版本撤回”；
    • 某部门所有工单默认优先级为中；
    • 某产品线必须引用最新季度文档。

    这些内容更像“用户画像与组织规则”，而不是原始聊天记录。把原始历史全部长期保留，只会让上下文越来越臃肿。

    # 9. 结构化输出：让模型结果进入工程系统

    如果模型输出只是自由文本，那么后续系统很难消费。上下文工程的一个关键目标，就是让模型输出变成下游系统可解析、可校验、可自动处理的结构。

    常见方式包括：

    • JSON Schema；
    • Markdown 表格；
    • 固定字段列表；
    • 枚举值限制；
    • 工具调用协议。

    例如，你希望模型判断一条工单是否应该升级，不要让它自由发挥，而应要求输出：

    {
  "should_escalate": true,
  "reason": "涉及支付失败且影响范围超过 20 个用户",
  "confidence": "high"
}
    1
    2
    3
    4
    5

    这样不仅更稳定，也更适合进入规则系统、工作流系统或监控系统。

    # 10. 从 Prompt Debug 走向系统 Debug

    很多 AI 应用调不稳，是因为团队总在做“文字级调试”：改一句话、换个语气、加个例子。但真正成熟的优化路径，应该是系统级调试：

    • 是模型选型不对，还是上下文不对？
    • 是检索结果不准，还是注入顺序不合理？
    • 是工具输出过长，还是状态管理混乱？
    • 是规则写得太模糊，还是任务目标本身不清楚？

    当你开始这样拆问题，就会发现很多所谓“Prompt 问题”，其实是数据流问题、状态问题、协议问题。

    # 11. 评测：上下文工程必须可验证

    上下文一旦复杂化，就不能靠体验做判断。建议至少建立三种评测：

    # 11.1 格式评测

    检查输出能否稳定满足协议，例如 JSON 是否合法、字段是否缺失。

    # 11.2 事实评测

    检查答案是否忠于提供证据，是否出现编造、越权推断、错误引用。

    # 11.3 任务评测

    检查用户目标是否完成，例如是否真的生成了符合要求的报告、是否正确使用了工具、是否在信息不足时发起追问。

    只有当评测能把失败样本稳定复现，你的上下文工程才进入可持续优化阶段。

    # 12. 常见反模式

    # 12.1 用一份 Prompt 解决所有场景

    不同任务的目标、约束、输出格式完全不同，强行复用一份 Prompt 只会越来越臃肿。

    # 12.2 把所有历史都塞进去

    历史不是资产，未经筛选的历史往往是噪声。系统应该保留“影响当前决策的信息”，而不是“所有曾经发生过的内容”。

    # 12.3 规则、材料、输出要求混在一起

    信息边界不清会让模型更容易漂移。最好把规则、任务、证据、格式分块组织。

    # 12.4 过度依赖魔法词

    像“请一步一步思考”“请认真分析”这类提示，有时有帮助，但它们不是系统能力的替代品。真正稳定的质量来自更好的上下文构造。

    # 13. 一个可执行的实践框架

    如果你要把现有 AI 应用从“Prompt 驱动”升级到“上下文驱动”，我建议按照以下顺序推进：

    1. 梳理系统所有输入来源：规则、检索、历史、工具、用户目标；
    2. 把输入按层分类，明确哪些是稳定模板，哪些是动态注入；
    3. 为每类上下文设置长度预算与优先级；
    4. 将输出协议结构化，减少自由文本；
    5. 建立最小评测集，覆盖常见失败样本；
    6. 把每次失败都归因到某一层，而不是统称“模型不稳定”。

    # 14. 总结

    提示词工程当然仍然重要，但它已经不再是大模型应用优化的全部。随着系统复杂度上升，真正决定质量上限的，是你如何组织上下文、如何分配注意力、如何约束输出、如何管理记忆、如何建立评测闭环。

    所以，与其问“这个 Prompt 怎么写更厉害”，不如问：当前这次任务，模型到底需要哪些信息、以什么结构拿到、哪些信息应该被排除在外。 当你开始这样思考时，你就已经进入上下文工程阶段了。

    # 15. 实操清单

    在优化你的 AI 应用前，建议先检查：

    • 规则、任务、证据、历史、格式是否分层？
    • 是否给上下文设置了优先级与长度预算？
    • 是否对检索与工具结果做了压缩和筛选？
    • 是否采用结构化输出？
    • 是否建立了失败样本库和回归评测？
    • 是否能区分“模型问题”和“上下文问题”？

    只要你把这些问题想清楚，系统稳定性通常会有明显提升。
