    # 构建可落地的 AI Agent：从工作流设计到评测治理

    预计阅读：18 分钟
    适合读者：希望把大模型从“能对话”推进到“能完成任务”的工程团队、技术负责人、独立开发者。

    很多团队在引入大模型时，第一步往往是做一个“会聊天”的助手。但当业务真正进入交付阶段，问题会迅速暴露：回答不稳定、工具调用混乱、上下文丢失、成本不可控、上线后无法评估。于是你会发现，真正有价值的不是一个“更会说话”的机器人，而是一个能围绕目标持续规划、调用工具、检查结果并在失败后恢复的 AI Agent。

    本文不讨论神秘的“全自动智能体幻觉”，而是聚焦可落地的工程方法：什么场景适合 Agent、一个靠谱的 Agent 应该有哪些组件、怎样从单体 Prompt 演化到可维护的系统、以及如何用评测、治理与观测让它跑在生产环境里。

    # 1. 什么时候该用 Agent，什么时候不该

    先说结论：不是所有 LLM 应用都应该升级成 Agent。

    如果你的需求只是“把输入转成固定格式输出”，例如文案润色、SQL 解释、接口文档总结，那么一个带结构化输出的普通工作流就足够。此时引入规划、多轮反思、工具编排，只会让延迟更高、结果更难预测。

    更适合 Agent 的场景通常有三个特点：

    1. 目标导向：用户给的是目标，不是完整步骤，例如“帮我整理本周所有客户会议，并输出风险清单”。
    2. 外部依赖多：需要查知识库、读工单、调接口、发消息、写数据库，而不是只靠模型自身记忆。
    3. 中间状态复杂：任务不是一步完成，而是要拆解、尝试、修正、确认后再继续。

    换句话说，Agent 解决的是“复杂任务编排”问题，不只是“自然语言生成”问题。

    # 2. Agent 与工作流、聊天机器人有什么区别

    很多概念容易混在一起，可以用一句话区分：

    • 聊天机器人：核心是对话体验。
    • 工作流应用：核心是固定流程。
    • AI Agent：核心是在约束下自主完成任务。

    一个典型的 Agent 系统至少要回答下面几个问题：

    • 当前目标是什么？
    • 我现在知道什么，还缺什么？
    • 下一步是继续思考，还是调用工具？
    • 工具结果是否可靠，是否需要二次确认？
    • 任务是否已经完成，是否需要把结果交给人审核？

    如果一个系统没有“状态推进”和“行动选择”，本质上就还不是 Agent，而只是一个带 Prompt 的应用。

    # 3.1 规划器

    规划器并不一定要是“超级智能的大脑”。在大多数业务里，规划器只要做好三件事：拆任务、选动作、判断结束条件。你可以把它理解为一个“会根据现场信息不断更新待办列表的调度器”。

    # 3.2 工具层

    工具决定了 Agent 的上限。没有工具，模型只能“猜”；有了工具，它才能“查、算、改、发”。工具的设计原则是：

    • 输入参数要严格定义；
    • 输出结构尽量稳定；
    • 权限边界必须清晰；
    • 每次调用都要可追踪；
    • 可失败，但不能悄悄失败。

    # 3.3 状态层

    很多 Agent 失败，不是模型不够强，而是状态管理太差。常见问题包括：多轮对话后忘记用户约束、工具结果没有归档、同一任务反复执行同一步。一个健康的状态层至少应记录：用户目标、关键约束、已完成步骤、工具调用结果、待确认事项、最终产物链接。

    # 4. 从单 Agent 到多 Agent：不要一开始就过度设计

    多 Agent 很诱人，因为听起来像“一个团队在协作”。但在工程上，多 Agent 不等于更好，很多时候只是把问题拆成了更多消息传递与状态同步问题。

    推荐演进路径如下：

    # 阶段一：单 Agent + 明确工具

    先用一个 Agent 完成闭环：理解目标、调用工具、输出结果。这一阶段最重要的是把工具定义好，把状态持久化做好，把失败场景跑通。

    # 阶段二：单 Agent + 子任务模板

    当任务逐渐复杂，可以引入固定子流程，例如“检索资料 → 提取事实 → 形成结论 → 输出摘要”。这里依然是一个 Agent 在驱动，只是内部使用了标准模板。

    # 阶段三：主 Agent + 专家能力

    只有当某些子任务具有明显不同的上下文、工具集或输出标准时，再拆出“专家角色”，例如：

    • 检索 Agent：只负责找资料与引用来源；
    • 分析 Agent：只负责对材料做对比推理；
    • 执行 Agent：只负责调用业务 API 落地操作；
    • 审核 Agent：只负责检查事实、权限与风险。

    # 阶段四：带人工审核的协同系统

    生产环境里最稳的，往往不是“纯自动 Agent”，而是“Agent + Human in the Loop”。尤其在财务、客服、法务、运维、代码修改这类高风险场景，人工确认不是降级，而是成熟系统的标配。

    # 5. 工具设计，决定 Agent 的可控性

    许多团队在模型上花了很多时间，却忽略了工具契约。实际上，工具接口就是 Agent 的操作系统。

    一个高质量工具应该做到：

    • 用自然语言好理解的 name 与 description；
    • 参数少而清晰，避免让模型填十几个字段；
    • 枚举值明确，时间格式统一；
    • 输出可解析，例如 JSON 对象而非自由文本；
    • 失败时返回可恢复错误，而不是只抛异常。

    举个例子，如果你要让 Agent 创建工单：

    {
  "tool": "create_ticket",
  "input": {
    "project": "billing",
    "priority": "high",
    "summary": "支付回调延迟导致订单状态未更新",
    "owner": "ops-team"
  }
}

    比起让模型“随便写一段说明，然后由后端猜字段”，这种结构化接口更容易复现与审计。

    # 6. 上下文管理：让 Agent 记住该记住的，忘掉该忘掉的

    上下文不是越长越好。很多项目失败的根因，是把聊天历史、工具结果、业务规则、用户附件一股脑塞进上下文，导致模型注意力被噪声稀释。

    更好的做法是分层：

    • 系统上下文：角色、原则、权限、输出约束；
    • 任务上下文：当前目标、用户要求、截止时间；
    • 工作记忆：已完成步骤、关键中间结果；
    • 外部知识：通过检索按需注入，而不是常驻；
    • 长期记忆：用户偏好、组织规则、术语字典。

    一个实用原则是：让上下文服务于下一步行动，而不是服务于“看起来很完整”。 任何无法影响下一步决策的信息，都应从主上下文里移出。

    # 7. 评测比 Prompt 更重要

    如果你上线后无法回答“这个 Agent 比上周更好了还是更坏了”，那它就还不具备可维护性。

    Agent 评测建议分三层：

    # 7.1 能力层评测

    关注单项能力是否可用，例如：

    • 是否能正确选择工具；
    • 是否能抽取正确字段；
    • 是否能按要求输出固定格式；
    • 是否能识别缺失信息并主动追问。

    # 7.2 任务层评测

    关注任务最终是否完成，例如“是否成功生成日报并发送到群里”“是否正确归并重复工单”。这类评测更接近真实业务价值。

    # 7.3 运营层评测

    关注上线后的真实指标：完成率、人工接管率、平均耗时、单任务成本、失败重试率、风险拦截率。很多看上去“回答更聪明”的优化，最终会让成本翻倍或失败率上升，因此必须用运营指标约束。

    # 8. 观测与日志：看不见，就改不好

    Agent 系统的日志不能只记录“请求成功/失败”。你真正需要的，是一份可回放的执行轨迹：

    • 用户输入与任务 ID；
    • 每一步规划内容；
    • 每次工具调用参数与结果；
    • 模型中间决策摘要；
    • 失败原因与重试策略；
    • 最终输出与人工修正。

    当你把这些轨迹记录下来后，很多问题就不再神秘。你会发现失败往往集中在很少几个点：工具描述模糊、上下文过长、缺少结束条件、对错误没有兜底。

    # 9. 成本与延迟控制：别把 Agent 变成奢侈品

    在 PoC 阶段，大家往往愿意容忍“慢一点、贵一点”。但一旦流量上来，成本就是架构问题。

    常见控制手段包括：

    • 优先使用更小、更快的模型处理路由、分类、抽取；
    • 只把复杂推理任务交给高能力模型；
    • 尽量减少无意义的自我反思轮次；
    • 对工具结果做缓存；
    • 对检索结果做摘要后再进入大模型；
    • 给每个任务设定最大步数与最大 token 预算。

    一个成熟团队不会问“最强模型是什么”，而会问“在满足 SLA 的前提下，哪种模型组合最划算”。

    # 10. 安全与治理：Agent 最危险的不是答错，而是做错

    如果模型只是生成一段错误文本，风险通常可控；但如果它调用外部系统执行删除、发送、审批、下单、改配置，风险就完全不同。

    因此，Agent 的安全治理至少要覆盖：

    • 权限最小化：不同角色只暴露必要工具；
    • 高风险动作确认：删除、付款、发外部消息必须人工确认；
    • 参数白名单：避免模型构造任意 SQL、任意 URL、任意命令；
    • 敏感信息脱敏：日志中隐藏 token、手机号、证件号；
    • 审计留痕：谁触发了什么任务，系统做了什么动作，要可追溯。

    # 11. 生产环境最常见的 6 个坑

    # 11.1 把 Agent 当成魔法

    很多失败项目从一开始就把模型当“会自动懂业务”的黑盒。真正有效的方法，是把业务规则显性化、工具契约结构化、状态管理产品化。

    # 11.2 一开始就上多 Agent

    多 Agent 会放大上下文同步、角色漂移和成本问题。单 Agent 跑不稳，拆成多个通常只会更糟。

    # 11.3 工具太多、太宽泛

    “万能工具”看起来省事，实际会让模型误用。一个职责单一、参数清晰的工具，远比一个大而全的工具更容易被正确调用。

    # 11.4 缺少停止条件

    很多 Agent 死循环，不是因为模型笨，而是因为系统没说清楚“什么时候算完成、什么时候该求助、什么时候必须结束”。

    # 11.5 没有失败路径

    如果工具报错后系统只会继续猜，最终只会制造更大的错误。正确做法是：失败时返回可解释状态，触发重试、降级或人工接管。

    # 11.6 只看 Demo，不看运营指标

    Demo 漂亮并不代表可上线。真正的门槛在于一周后、一个月后、规模化之后是否依然稳定。

    # 12. 一个可执行的落地路线图

    如果你的团队准备建设第一个 Agent 系统，可以按下面四步推进：

    # 第一步：找准高价值、低风险任务

    优先选择文档整理、知识检索、工单辅助、内部运营分析这类可回滚任务，而不是直接做自动审批、自动改生产配置。

    # 第二步：先把工具层做好

    把最关键的 3 到 5 个工具定义清楚，明确输入、输出、错误码、权限要求。工具稳定后，模型效果通常会自然提升一大截。

    # 第三步：建立最小评测集

    找 30 到 50 个真实任务样本，覆盖成功案例、边界案例、失败案例。每次改 Prompt、模型或工具描述时，都要回归一次。

    # 第四步：把人工审核设计进系统

    不要把人工审核视为“系统不够智能”的补丁，而要把它当成生产控制面的一部分。越高价值、越高风险的任务，越应该具备人工兜底能力。

    # 13. 总结

    AI Agent 的核心价值，不在于让模型“表现得像人”，而在于让系统围绕目标持续推进任务。从工程视角看，靠谱的 Agent 不是单纯依赖大模型参数能力，而是依赖四个字：结构、约束、反馈、治理。

    如果你只能记住一件事，我建议是：先把 Agent 当成一个有状态、有工具、有边界的业务系统来设计，再把大模型放进去。 这样做出来的系统，才更接近产品，而不是演示。

    # 14. 实操清单

    在开始你的 Agent 项目前，至少确认下面这些问题：

    • 这个任务真的需要自主规划吗？
    • 哪些步骤必须调工具，哪些步骤只需要生成？
    • 状态如何持久化，失败后如何恢复？
    • 哪些动作必须人工确认？
    • 是否有最小评测集和回归机制？
    • 是否能追踪每一步决策与成本？

    当这些问题都有答案时，你的 Agent 才真正具备上线资格。
